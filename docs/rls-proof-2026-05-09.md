## Bug surfaced during proof, fixed

Migration 0096 shipped two write-side RLS policies (`provider_update_enrolments`, `provider_insert_own_disputes`) that depend on table-level GRANTs that did not exist. PostgreSQL evaluates GRANT before RLS, so denied GRANT short-circuits the policy entirely with `42501 permission denied`. Net effect: the portal Server Action `markOutcomeAction` had been silently unable to persist outcomes since shipping in Session 37, despite the handoff marking it owner-tested. Confirmed by inspecting `crm.enrolments.updated_at` for the demo provider — every row carried the seed-time timestamp, no Charlotte-driven UPDATEs had landed.

Fix: migration 0108 grants `UPDATE ON crm.enrolments` and `INSERT ON crm.disputes` to `authenticated`. Row scope continues to be enforced by the 0096 policies (the proof above re-runs after 0108 applied, with cross-tenant write tests still returning zero rows / RLS rejections).

## What this proves and what it doesn't

**Proves:**
- Helper function correctly identifies the provider for an authenticated session, returns NULL outside the portal-enabled scope.
- Cross-tenant SELECT, UPDATE, INSERT through supabase-js are all blocked at the row level.
- Audit writes attribute to the actual caller, not whatever `actor_provider_id` the caller passes via context.
- The portal_enabled flag is the single switch for cutting off provider portal access without touching policies.

**Does not prove:**
- WebAuthn ceremony correctness (separate concern, not RLS-relevant).
- Any path that uses the service-role key from the API routes (those bypass RLS by design and are gated by app code).
- Long-running session behaviour (cookie refresh, session expiry, etc.).
- Audit chain integrity across many writes (covered by `audit.actions` schema invariants, not by this proof).
